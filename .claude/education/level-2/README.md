# Level 2: 実践

> より複雑なIssueを書き、エスカレーションに対応しよう

---

## このレベルで学ぶこと

1. 複数要件のIssue作成
2. エスカレーション対応
3. 品質の確認方法

---

## 課題 2-1: API実装Issue

### 目標
複数の要件を含むIssueを書く

### やること

1. 以下のIssueを作成：

```bash
gh issue create --title "機能: ユーザー管理API" --body "## 概要
ユーザーを管理するAPIを作りたい

## やりたいこと
- ユーザー一覧取得 GET /users
- ユーザー作成 POST /users
- ユーザー取得 GET /users/:id
- ユーザー削除 DELETE /users/:id

## 制約
- インメモリで保存（DBは使わない）
- TypeScript strict mode

## テスト
- 各エンドポイントのテストを書く"
```

2. 処理を実行してPRを待つ

3. PRの内容を確認：
   - 4つのエンドポイントがあるか
   - テストが書かれているか

### 成功条件
- [ ] 複数要件のIssueが書けた
- [ ] 全ての要件が実装された
- [ ] テストが含まれている

---

## 課題 2-2: エスカレーション体験

### 目標
エスカレーションに対応する経験を積む

### やること

1. 意図的に曖昧なIssueを作成：

```bash
gh issue create --title "機能: 認証機能" --body "## 概要
ユーザー認証ができるようにしたい"
```

2. 処理を実行

3. Agentから質問が来る（エスカレーション）：
   - 「認証方式は何にしますか？」
   - 「セッションの有効期限は？」

4. 質問に答える

5. Agentが続行してPRを作成

### 成功条件
- [ ] エスカレーションを受けた
- [ ] 判断を伝えた
- [ ] Agentが続行してPRを作った

### 学び
```
エスカレーション = Agentが人間に判断を委ねる
これは正常な動作。怖がらなくてOK。
```

---

## 課題 2-3: 品質改善

### 目標
品質スコアを意識してIssueを書く

### やること

1. 具体的なIssueを作成：

```bash
gh issue create --title "機能: バリデーション関数" --body "## 概要
入力値を検証する関数群を作りたい

## やりたいこと
- isEmail(str): メールアドレス形式か判定
- isURL(str): URL形式か判定
- isEmpty(str): 空文字・null・undefined判定

## 制約
- 正規表現を使用
- エッジケースを考慮

## テスト要件
- 正常系・異常系の両方をテスト
- カバレッジ80%以上"
```

2. PRの品質を確認：
   - TypeScriptエラー: 0件
   - ESLintエラー: 0件
   - テストカバレッジ: 80%+

### 成功条件
- [ ] 具体的な制約を書けた
- [ ] 品質スコア80点以上のPRができた
