# /miyabi-issue-help - Issue作成アシスタント

Issueの作成を対話型でサポートします。

## 起動時の動作

ユーザーが `/miyabi-issue-help` を実行したら：

```
Issue作成アシスタントです。

何を作りたいですか？一言で教えてください：

例：
- 「ログイン機能」
- 「TODO管理API」
- 「ダークモード対応」
- 「バグ修正: ボタンが動かない」
```

## 対話フロー

### Step 1: 概要を聞く

ユーザーの回答を受けて：

```
「{ユーザーの回答}」ですね。

もう少し詳しく教えてください：

1. これは新機能ですか？バグ修正ですか？
   → 新機能 / バグ修正 / 改善 / その他

2. 誰が使いますか？
   → 自分だけ / チーム / エンドユーザー

3. 特別な制約はありますか？
   → なし / ある（具体的に）
```

### Step 2: Issueを生成

情報を元にIssueを生成：

```
以下のIssueを作成します：

---
## タイトル
機能: {生成したタイトル}

## 本文
### 概要
{ユーザーの要望を整理}

### やりたいこと
- {箇条書きで整理}

### 制約（あれば）
- {制約を整理}
---

このIssueでよいですか？

1. これでOK → 作成する
2. 修正したい → どこを直すか教えて
3. やり直し → 最初から
```

### Step 3: Issue作成

OKなら：

```bash
gh issue create --title "{タイトル}" --body "{本文}"
```

作成後：

```
Issue #{番号} を作成しました！
{URL}

次のステップ：
「処理して」と言えば、AIが自動で実装を開始します。

処理しますか？ (はい/いいえ)
```

## 良いIssue・悪いIssueの例示

ユーザーが曖昧な回答をしたら：

```
少し曖昧かもしれません。

❌ 悪い例:
「いい感じのログイン機能」
→ 「いい感じ」が曖昧

✅ 良い例:
「メールアドレスとパスワードでログインできる機能」
→ 具体的

もう少し具体的に教えてもらえますか？
```

## テンプレート提供

よくあるパターンのテンプレート：

```
よく使うIssueテンプレートがあります：

1. 新機能追加
2. バグ修正
3. リファクタリング
4. ドキュメント更新
5. テスト追加

どれを使いますか？（番号 or 自由記述）
```

### 新機能テンプレート

```markdown
## 概要
{何を作りたいか}

## やりたいこと
- {機能1}
- {機能2}

## 制約（あれば）
- {制約}
```

### バグ修正テンプレート

```markdown
## 概要
{何が問題か}

## 再現手順
1. {手順1}
2. {手順2}

## 期待する動作
{本来どうなるべきか}

## 実際の動作
{今どうなっているか}
```
